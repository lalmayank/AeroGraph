# OpenTelemetry Bridge End-to-End Workflows

This guide outlines how to use the AeroGraph OTLP Bridge across end-to-end interoperability workflows.

## Workflow 1: AeroGraph → OTLP → Jaeger

To export traces generated natively in AeroGraph into an external observability platform like Jaeger:

1. **Emit Traces**: Use the `@aerograph/sdk` or `aerograph-sdk` to generate and emit traces to the AeroGraph Collector.
2. **Fetch Traces**: Pull trace events from the Collector (`GET /v1/traces/:traceId`).
3. **Export to OTLP**: Pass the trace events through the Export Bridge:
   ```typescript
   import { exportEventsToOtlp } from '@aerograph/otel';
   const otlpRequest = exportEventsToOtlp(events);
   ```
4. **Push to Jaeger**: POST the resulting `otlpRequest` JSON directly to Jaeger's HTTP ingestion endpoint. The hierarchy, links, and specific `aerograph.*` attributes will appear in the Jaeger UI seamlessly.

## Workflow 2: OTLP → Collector → AeroGraph

To ingest traces generated by an external OpenTelemetry system (e.g., standard LangChain OTel exporters) into the AeroGraph Collector:

1. **Point External SDK to AeroGraph**: Configure your external application's OTLP Trace Exporter (HTTP) to point to the AeroGraph Collector:
   `http://localhost:4317/v1/otlp/traces`
2. **Execute Traces**: Run your external application. The Collector will catch incoming OTLP spans, convert them dynamically, and persist them.
3. **View in AeroGraph**: Open the AeroGraph UI (`http://localhost:5173`). The external spans will appear alongside native traces. Heuristics will automatically label generic `gen_ai.chat` spans as `response` events.

## Workflow 3: AeroGraph → OTLP → AeroGraph (Roundtrip)

A complete roundtrip proves the bridge's lossless translation.

1. Create a `TraceEvent` natively in AeroGraph.
2. Export the event to an `OtlpSpan` via `exportEventToOtlpSpan`.
3. Import the generated `OtlpSpan` back into a `TraceEvent` using `importOtlpSpanToEvent`.
4. Validate that the original topology and metadata (including `traceId`, `spanId`, `parentSpanId`, and semantic payload) are 100% preserved. 
   > **Note**: AeroGraph uses `aerograph.kind` within the OTLP attributes to ensure perfect roundtrip mapping without information loss.

## Trace Correlation (Stitching)

You can correlate AeroGraph traces directly with existing OpenTelemetry traces. Simply extract the `traceId` and `spanId` from an incoming W3C Trace Context and pass them into the `FlightRecorder`:

```python
from aerograph_sdk import FlightRecorder
recorder = FlightRecorder(trace_id="external_w3c_trace_id", actor={"id": "app"})
recorder.note(parent_span_id="external_w3c_span_id", text="Execution started")
```
This stitches your AeroGraph traces seamlessly under the existing external OTel root span in tools like Jaeger or Datadog.
